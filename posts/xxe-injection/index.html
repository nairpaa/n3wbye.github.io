Pada kesempatan kali ini saya akan mencoba menjelaskan apa itu XXE injection, beberapa contoh umum XXE injection, menemukan dan mengeksploitasi berbagai jenis XXE injection, dan menyimpulkan cara mencegah serangan XXE injection.

1. Apa itu XML external enitity (XXE) injection?#

XML external entity injection (dikenal juga sebagai XXE) adalah kerentanan keamanan web yang memungkinkan penyerang untuk mengganggu pemrosesan aplikasi data XML. Seringkali memungkinkan penyerang untuk melihat file yang berada di dalam server.

Dalam beberapa situasi, penyerang dapat meningkatkan serangan XXE untuk membahayakan server yang mendasarinya atau infrastruktur back-end lainnya, dengan memanfaatkan kerentanan XXE untuk melakukan serangan server-side request forgery (SSRF) .

2. Bagaimana kerentanan XXE dapat muncul?#

Beberapa aplikasi menggunakan format XML untuk mengirimkan data antara browser dan server. Aplikasi yang menggunakan cara ini hampir selalu menggunakan library standar atau platform API untuk memproses data XML pada server.

Kerentanan XXE muncul karena spesifikasi XML berisi berbagai jenis fitur yang berpotensi berbahaya, dan parser standar mendukung fitur ini meskipun biasanya tidak digunakan oleh aplikasi.

  • Pelajari tentang XML format, DTD dan external entitas.

XML external entities adalah jenis dari custom XML entity yang nilainya ditentukan dimuat dari luar DTD tempate mereka dideklarasikan. External entities sangat menarik dari perspektif keamanan karena memungkinkan entitas untuk didefinisikan berdasarkan konten jalur file atau URL.

3. Apa saja jenis serangan XXE?#

Ada berbagai jenis serangan XXE, diantaranya:

  • Exploiting XXE to retrieve files, di mana sebuah external entity didefinisikan berisi konten file, dan dikembalikan dalam respons aplikasi.
  • Exploiting XXE to perform SSRF attacks, dimana sebuah external entity didefinisikan berdasarkan URL ke sistem back-end.
  • Exploiting blind XXE exfiltrate data out-of-band, di mana data sensitif ditransmisikan dari server aplikasi ke sistem yang dikendalikan penyerang.
  • Exploiting blind XXE to retrieve data via error messages, di mana penyerang dapat memicu pesan kesalahan penguraian yang berisi data sensitif.

3.1. Memanfaatkan XXE untuk mengambil file#

Untuk melakukan serangan injeksi XXE yang mengambil file dari server, kita perlu memodifikasi XML yang dikirimkan dengan dua cara:

  • Introduce (atau edit) elemen DOCTYPE yang mendefinisikan external entity yang berisi PATH ke file.
  • Edit nilai data dalam XML yang dikembalikan dalam respons aplikasi, untuk menggunakan external entity yang ditentukan.

Misalnya, aplikasi shopping memeriksa stok produk dengan mengirimkan XML berikut ke server:

<?xml version="1.0" encoding="UTF-8"?>
<stockCheck><productId>381</productId></stockCheck>

Aplikasi tidak melakukan pengamanan khusus terhadap serangan XXE, sehingga kita dapat mengeksploitasi kerentanan XXE untuk mengambil file /etc/passwd dengan mengirimkan payload XXE berikut:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
<stockCheck><productId>&xxe;</productId></stockCheck>

Payload XXE ini mendefinisikan external entity &xxe; yang nilainya adalah isi file /etc/passwd dan menggunakan entitas dalam nilai productId. Ini menyebabkan respons aplikasi menampilkan konten file tersebut:

Invalid product ID: root❌0:0:root:/root:/bin/bash
daemon❌1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin❌2:2:bin:/bin:/usr/sbin/nologin
...

Cacatan:

Kerentanan XXE di dunia nyata, seringkali akan ada sejumlah besar nilai data dalam XML yang dikirimkan, salah satunya mungkin digunakan dalam respons aplikasi. Untuk menguji kerentanan XXE secara sistematis, kita biasanya perlu menguji setiap node data dalam XML satu per satu, dengan memanfaatkan entitas yang kita tetapkan dan melihat apakah itu muncul dalam respons.

3.2. Memanfaatkan XXE untuk melakukan serangan SSRF#

Selain pengambilan data sensitif, dampak utama lain dari serangan XXE adalah dapat digunakan untuk melakukan serangan server-side request forgery (SSRF).

Ini adalah kerentanan yang berpotensi serius di mana aplikasi sisi server dapat diinduksi untuk membuat request HTTP ke URL apa pun yang dapat diakses server.

Untuk mengeksploitasi kerentanan XXE untuk melakukan serangan SSRF, kita perlu mendefinisikan external XML entity menggunakan URL yang ingin kita targetkan, dan menggunakan entitas yang ditentukan dalam nilai data.

Jika kita dapat menggunakan entitas yang ditentukan dalam nilai data yang dikembalikan dalam respons aplikasi, maka kita akan dapat melihat respons dari URL dalam respons aplikasi, dan dengan demikian mendapatkan interaksi dua arah dengan sistem back-end.

Jika tidak, maka kita hanya akan dapat melakukan serangan Blind SSRF (yang masih memiliki konsekuensi kritis).

Dalam contoh XXE berikut, external entity akan menyebabkan server membuat request HTTP back-end ke sistem internal dalam infrastruktur organisasi:

<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://internal.vulnerable-website.com/"> ]>

3.3. Kerentanan Blind XXE#

Banyak contoh kerentanan blind XXE. Ini berarti bahwa aplikasi tidak mengembalikan nilai dari external entity yang ditentukan dalam responsnya, sehingga melihat isi file secara langsung tidak dimungkinkan.

Kerentanan blind XXE masih dapat dideteksi dan dieksploitasi, tetapi diperlukan teknik yang lebih canggih. Terkadang kita dapat menggunakan teknik out-of-band untuk menemukan kerentanan dan mengeksploitasinya untuk mengekstrak data. Dan kita terkadang dapat memicu parsing errors XML yang mengarah pada pengungkapan data sensitif dalam pesan kesalahan.

4. Menemukan attack surface tersembunyi untuk XXE injection#

Attack surface untuk kerentanan XXE injection terlihat jelas dalam banyak kasus, karena lalu lintas HTTP normal aplikasi menyertakan request yang berisi data dalam format XML.

Dalam kasus lain, attack surface kurang terlihat. Namun, jika kita mencari di tempat yang tepat, kita akan menemukan serangan XXE muncul di request yang tidak berisi XML apa pun.

4.1. XInclude attacks#

Beberapa aplikasi menerima data yang dikirimkan klien, menyematkannya di sisi server ke dalam dokumen XML, lalu mengurai dokumen.

Contoh dari hal ini terjadi ketika data yang dikirimkan klien ditempatkan ke dalam request SOAP back-end, yang kemudian diproses oleh SOAP back-end service.

Dalam situasi ini, kita tidak dapat melakukan serangan XXE klasik, karena kita tidak mengontrol seluruh dokumen XML sehingga tidak dapat menentukan atau memodifikasi elemen DOCTYPE. Namun, kita mungkin dapat menggunakan XInclude sebagai gantinya.

XInclude adalah bagian dari spesifikasi XML yang memungkinkan dokumen XML dibuat dari sub-dokumen. Kita dapat menempatkan serangan XInclude dalam nilai data apa pun dalam dokumen XML, sehingga serangan dapat dilakukan dalam situasi di mana kita hanya mengontrol satu item data yang ditempatkan ke dalam dokumen XML sisi server.

Untuk melakukan serangan XInclude, kita perlu merujuk namespace XInclude dan menyediakan PATH ke file yang ingin Anda sertakan. Sebagai contoh:

<foo xmlns:xi="http://www.w3.org/2001/XInclude">
<xi:include parse="text" href="file:///etc/passwd"/></foo>

4.2. XXE attacks via file upload#

Beberapa aplikasi memungkinkan pengguna untuk mengunggah file yang kemudian diproses di sisi server. Beberapa format file umum menggunakan XML atau berisi subkomponen XML. Contoh format berbasis XML adalah format dokumen kantor seperti DOCX dan format gambar seperti SVG.

Misalnya, aplikasi mungkin mengizinkan pengguna untuk mengunggah gambar, dan memproses atau memvalidasinya di server setelah diunggah. Bahkan jika aplikasi mengharapkan untuk menerima format seperti PNG atau JPEG, library pemrosesan gambar yang digunakan mungkin mendukung gambar SVG. Karena format SVG menggunakan XML, penyerang dapat mengirimkan gambar SVG berbahaya sehingga mencapai permukaan serangan tersembunyi untuk kerentanan XXE.

4.3. XXE attacks via modified content type#

Sebagian besar request POST menggunakan content-type default yang dihasilkan oleh forms HTML, seperti application/x-www-form-urlencoded. Beberapa situs web berharap untuk menerima request dalam format ini tetapi akan mentolerir jenis konten lain, termasuk XML.

Misalnya, jika request normal seperti berikut:

POST /action HTTP/1.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 7

foo=bar

Maka kita mungkin dapat mengirimkan request berikut, dengan hasil yang sama:

POST /action HTTP/1.0
Content-Type: text/xml
Content-Length: 52

<?xml version="1.0" encoding="UTF-8"?><foo>bar</foo>

Jika aplikasi mentolerir request yang berisi XML dalam message body, dan mem-parsing body content sebagai XML, maka kita dapat mencapai attack surface XXE yang tersembunyi hanya dengan memformat ulang request untuk menggunakan format XML.

5. Cara mencari dan menguji kerentanan XXE#

Sebagian besar kerentanan XXE dapat ditemukan dengan cepat dan andal menggunakan Burp suite Scanner.

Pengujian secara manual untuk kerentanan XXE umumnya melibatkan:

  • Menguji pengambilan file dengan mendefinisikan external entity berdasarkan file sistem operasi yang digunakan dan menggunakan entitas tersebut dalam data yang dikembalikan dalam respons aplikasi.
  • Menguji kerentanan blind XXE dengan mendefinisikan external entity berdasarkan URL ke sistem yang kita kontrol, dan memantau interaksi dengan sistem itu. Burp Collaborator client sangat cocok untuk tujuan ini.
  • Menguji vulnerable inclusion untuk data non-XML yang disediakan pengguna dalam dokumen XML sisi server dengan menggunakan serangan XInclude untuk mencoba mengambil file yang ada di dalam server.

6. Cara mencegah kerentanan XXE#

Hampir semua kerentanan XXE muncul karena parsing library XML aplikasi mendukung fitur XML yang berpotensi berbahaya yang tidak diperlukan atau tidak ingin digunakan oleh aplikasi. Cara termudah dan paling efektif untuk mencegah serangan XXE adalah dengan menonaktifkan fitur tersebut.

Secara umum, cukup menonaktifkan resolution of external entities dan menonaktifkan dukungan untuk XInclude. Ini biasanya dapat dilakukan melalui opsi konfigurasi atau dengan mengesampingkan konfigurasi default secara terprogram. Lihat dokumentasi untuk parsing library XML atau API yang digunakan untuk detail tentang cara menonaktifkan fitur yang tidak diperlukan.

7. Closing#

Semoga penjelasan tentang XXE injection ini bisa teman-teman pahami dengan baik. Happy Hacking!
